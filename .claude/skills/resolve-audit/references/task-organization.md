# Task 정리 노하우

감사 결과물의 findings를 효율적으로 정리하고 실행 순서를
결정하는 방법.

## 1단계: 의존성 그래프 산출

분류 테이블을 만든 직후, findings 사이의 의존성을 파악한다.
의존성이란 "X를 먼저 처리하지 않으면 Y를 처리할 수 없거나,
Y의 난이도가 현저히 올라가는 관계"를 뜻한다.

흔한 의존 패턴:

- **파일 의존**: B-1(god file 분리)을 해야 A-3(분리된 모듈에
  에러 처리 추가)이 가능하다.
- **인터페이스 의존**: A-1(API 시그니처 변경)이 완료되어야
  B-4(해당 API를 사용하는 코드 정리)가 가능하다.
- **테스트 의존**: B-2(테스트 인프라 정리)가 완료되어야
  A-2(새 테스트 추가)가 안정적이다.

의존성을 찾는 실용적 방법:

```
각 finding의 수정 대상 파일 목록을 나열한다.
파일이 겹치는 finding 쌍을 찾는다.
겹치는 쌍 중, 한쪽이 구조를 변경하고(B)
다른 쪽이 동작을 변경한다면(A), B → A 순서로 고정한다.
```

## 2단계: 실행 순서 결정

의존성 그래프를 바탕으로 위상 정렬(topological sort)한다.
같은 레벨의 항목들은 아래 우선순위로 정렬:

1. **Category A (TDD) 먼저**: 행위 변경이 가장 위험하므로
   조기에 처리하여 피드백을 빨리 받는다.
2. **고severity 먼저**: Critical > High > Medium > Low.
3. **작은 effort 먼저** (같은 severity 내): S 항목을 먼저
   처리하면 진행감이 생기고, 초반 커밋들이 코드베이스
   상태를 개선하여 후반 큰 작업이 쉬워진다.
4. **파일 클러스터링**: 같은 파일을 건드리는 항목을
   연속으로 배치하여 컨텍스트 전환을 줄인다.

결과물 형식:

```
Execution Order:
1. A-1 (Critical, S) — 의존성 없음
2. B-1 (High, M) — 의존성 없음
3. A-3 (High, S) — depends on B-1
4. B-2 (Medium, S) — 의존성 없음, B-4와 파일 겹침
5. B-4 (Medium, M) — depends on A-1, B-2 이후 배치
...
```

## 3단계: 배치 그룹 구성

서브에이전트에 할당하기 위해 독립 그룹으로 묶는다.

규칙:
- 같은 그룹의 항목들은 파일 겹침이 없어야 한다.
- 의존성이 있는 항목은 같은 그룹에 넣되 순서를 고정한다.
- 한 그룹의 크기는 effort 합산 기준으로 균등하게 분배한다.
  (S=1, M=3, L=8 정도의 가중치)

예시:

```
Group-A (직렬, TDD 전담):
  A-1 (S) → A-2 (M) → A-3 (S)   [총 effort: 5]

Group-B1 (병렬 가능):
  B-1 (M), B-3 (S)               [총 effort: 4]

Group-B2 (병렬 가능):
  B-2 (S) → B-4 (M)              [총 effort: 4, B-2→B-4 직렬]

Group-C (병렬 가능):
  C-1 (S), C-2 (S), C-3 (S)      [총 effort: 3]
```

## 4단계: 체크포인트 전략

장시간 세션에서 중간 검증을 삽입하는 기준:

- **5개 커밋마다** 전체 테스트 스위트를 실행한다.
  개별 커밋 후에도 테스트를 돌리지만, 5개 간격의
  전체 실행은 cross-finding 회귀를 잡기 위한 것이다.
- **Phase 전환마다** (1→2, 2→3) 전체 테스트 + 린터를
  돌리고, memory를 갱신한다 (memory-protocol.md 참조).
- **재분류 발생 시** 즉시 실행 순서를 재계산한다.
  B→A 전환은 의존성 그래프를 변경할 수 있다.

## 5단계: 지뢰 사전 탐지

처리를 시작하기 전에 다음을 확인한다:

- **기존 테스트 스위트가 통과하는가?** 실패하는 테스트가
  있으면, 그것이 감사 항목의 원인인지 무관한 것인지
  판단한다. 무관하면 해당 테스트를 skip 마킹하고 사유를
  기록한다. 감사 항목의 원인이면 해당 항목의 priority를
  올린다.
- **브랜치가 main에서 갈라진 시점 이후 main에 변경이
  있었는가?** 있으면 rebase를 먼저 한다.
- **감사 이슈에 "Open questions" 섹션이 있는가?**
  있으면 해당 질문을 사용자에게 먼저 확인한다.
  답이 없는 상태에서 관련 항목을 처리하면 재작업이
  발생한다.
