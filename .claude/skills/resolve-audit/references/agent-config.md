# 에이전트 구성 지침 (일반 개발방식)

Dynamic Workflow를 사용하지 않고, Task 도구로 서브에이전트를
수동 배분할 때의 구성 방법.

## 모델 및 Effort 원칙

**SOTA 모델 고정**: 메인과 서브에이전트 모두 사용 가능한
최고 성능 모델을 사용한다. 현시점 기준 Opus 4.8.
서브에이전트에 Sonnet을 배정하여 비용을 아끼려는 유혹이
있으나, 리팩토링과 TDD는 추론 깊이에 민감한 작업이다.
Sonnet이 놓친 행위 변경을 메인이 리뷰에서 잡아야 하면
오히려 재작업 비용이 더 크다.

**Effort 복제**: 메인이 xhigh이면 서브도 xhigh.
서브에이전트의 effort를 낮추면 얕은 TDD 루프
(Red를 충분히 구체적으로 작성하지 않거나, Green에서
과도한 코드를 작성하는 것)가 발생한다.

## 역할 분배

```
메인 모델 (Opus 4.8, xhigh)
├── Phase 0: 분류, 의존성 분석, 사용자 확인
├── 서브에이전트 디스패치
├── 각 서브에이전트 결과의 코드 리뷰
├── 재분류 판단 (B→A 전환 등)
├── Phase 4: 최종 검증, PR 생성
│
서브에이전트 (Opus 4.8, xhigh)
├── 할당된 findings 처리 (A/B/C 프로토콜)
├── 커밋 생성
└── 이상 발생 시 메인에 보고
```

## 병렬 디스패치 대상

서브에이전트를 병렬로 띄울 수 있는 것은 Category B와 C에
한정된다. 조건:

- **파일 겹침 없음**: 두 finding이 같은 파일을 수정하면
  같은 서브에이전트에 배정하거나 직렬로 처리.
- **의존성 없음**: B-2의 결과가 B-5에 영향을 주면 직렬.

Category A는 반드시 하나의 에이전트가 직렬로 처리한다.
아래 TDD 담당 섹션 참조.

## TDD 담당: 작업 에이전트

**결정: 작업 에이전트(서브)가 TDD를 실행한다.
메인 모델은 완료된 커밋을 리뷰한다.**

근거:

1. TDD의 Red-Green-Refactor는 코드 작성과 테스트 작성이
   한 루프 안에서 교대한다. Red에서 실패하는 테스트를
   작성하려면 현재 코드 상태를 정확히 알아야 하고,
   Green에서 최소 코드를 작성하려면 방금 작성한 테스트를
   보고 있어야 한다. 이 루프를 두 에이전트로 쪼개면
   컨텍스트 전달 오버헤드가 루프의 즉시성을 파괴한다.

2. 메인 모델이 TDD를 직접 수행하면, 모든 Category A
   항목이 직렬로 메인에 묶여서 B/C의 병렬 처리를 활용할
   수 없다. 메인이 A를 처리하는 동안 아무도 B/C를 못 한다.

3. 서브에이전트에 TDD를 위임하면, 메인은 그 시간 동안
   B/C 서브에이전트를 감독하고, A 서브에이전트의 커밋이
   올라오는 대로 리뷰할 수 있다.

따라서 구성:

```
메인 모델
├── 디스패치 + 감독 + 리뷰
│
├── SubAgent-A (직렬, TDD 전담)
│   A-1: Red → Green → Refactor → commit
│   A-2: Red → Green → Refactor → commit
│   ...
│
├── SubAgent-B1 (병렬 그룹 1)
├── SubAgent-B2 (병렬 그룹 2)
├── SubAgent-C1 (병렬)
└── ...
```

## 메인 모델의 코드 리뷰 프로토콜

각 서브에이전트가 커밋을 생성할 때마다, 메인 모델이
해당 커밋의 diff를 리뷰한다. 리뷰는 서브에이전트의
작업과 비동기로 진행할 수 있다.

리뷰 체크리스트:

**Category A 커밋 리뷰**:
- [ ] 테스트가 행위를 검증하는가 (구현 세부가 아닌)
- [ ] 테스트명이 행위를 영어/한국어 문장으로 서술하는가
- [ ] Green에서 최소한의 코드만 작성했는가
- [ ] Refactor에서 테스트가 여전히 통과하는가
- [ ] 전체 테스트 스위트가 통과하는가
- [ ] 커밋 메시지에 finding ID가 포함되어 있는가

**Category B 커밋 리뷰**:
- [ ] 공개 인터페이스가 변경되지 않았는가
- [ ] 새 테스트가 추가되지 않았는가 (B에서 금지)
- [ ] 전체 테스트 pass count가 보존되었는가
- [ ] 할당된 파일 범위를 벗어나지 않았는가

**Category C 커밋 리뷰**:
- [ ] 런타임 코드 변경이 섞이지 않았는가
- [ ] 린터/컴파일러가 통과하는가

리뷰에서 문제를 발견하면:
1. 해당 커밋을 revert
2. 서브에이전트에 구체적 수정 지시 전달
3. 수정된 커밋을 재리뷰

리뷰 통과 시 다음 항목으로 진행.
